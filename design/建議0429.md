# SightingAssistantTool 改動建議 — 提升 Extension 整合體驗

> 日期：2026-04-29  
> 來源：Chrome Extension + Bridge 開發與測試過程中發現的實際問題  
> 目標：讓 Extension 自動化場景下的分析流程更順暢、更可靠

---

## 中文版

### 建議 1：移除 subprocess batch 中的 `pause`

**問題：**  
`sherlog_subprocess.py` 和 `displaydebugger_subprocess.py` 產生的 batch 檔案結尾有 `pause`，會讓子視窗卡住等使用者按鍵。CLI 手動操作時有用，但在自動化 / Extension 場景下，子視窗不會自動關閉，導致整個分析流程卡住。

**影響位置：**
- `src/sherlog_subprocess.py` 第 116 行
- `src/displaydebugger_subprocess.py` 對應位置

**建議改法：**  
移除 `pause`，或改為環境變數控制：
```batch
if "%GNAI_AUTOMATED%"=="" pause
```
這樣 CLI 使用者仍可手動檢視輸出，而 Extension 設定 `GNAI_AUTOMATED=1` 後子視窗會自動關閉。

**工作量：** 5 分鐘  
**效果：** 子視窗自動關閉，Bridge 不再需要 taskkill workaround

---

### 建議 2：Phase 1 拆成兩步 — 快速資料先輸出，耗時 tool 後執行

**問題：**  
目前 `sighting_assistant.yaml` 的 prompt 讓 LLM 一口氣呼叫所有 Phase 1 tools（包含 `sherlog_sync`、`displaydebugger`），等**全部完成**後才開始輸出第一行回應。Sherlog 分析可能需要 1–5 分鐘，使用者在這段時間只看到 loading spinner，完全沒有內容。

**影響位置：**
- `assistants/sighting_assistant.yaml` prompt 區段

**建議改法：**  
在 prompt 中明確將 Phase 1 拆為兩步：

- **Step A（快速，10–20 秒）：** 呼叫 `sighting_read_article`、`sighting_attachments`、`sighting_similarity_search`、`sighting_rag_search` → 完成後**立即輸出初步摘要**（HSD 標題、分類、元件、附件列表、相似 HSD、DFD checklist）
- **Step B（耗時，1–5 分鐘）：** 呼叫 `sighting_sherlog_sync`、`sighting_gop_analyzer`、`sighting_displaydebugger` → 完成後追加深度分析結果

Prompt 中需加入強制指令：`YOU MUST output the preliminary summary BEFORE calling Step B tools. DO NOT batch all tools together.`

**工作量：** 30 分鐘（只改 prompt 文字，不動 tool code）  
**效果：** 使用者 10–20 秒就能看到 HSD 基本資訊，不需等 Sherlog 跑完

---

### 建議 3：互動問題加結構化標記

**問題：**  
LLM 問使用者選擇時（如選擇附件分析、是否執行 DisplayDebugger），回應格式不固定。Extension 目前只能用 regex 猜測編號列表或 Yes/No 問題，準確率約 80%，偶爾會產生錯誤的按鈕或漏掉按鈕。

**影響位置：**
- `assistants/sighting_assistant.yaml` prompt 區段

**建議改法：**  
在 prompt 中要求 LLM 用固定標記包裹互動問題：

多選題：
```
[USER_SELECT]
1) SystemScopeTool.zip
2) crash_dump.dmp
3) All
[/USER_SELECT]
```

確認題：
```
[USER_CONFIRM]
Would you like to run DisplayDebugger analysis on the display logs?
[/USER_CONFIRM]
```

Extension 透過偵測 `[USER_SELECT]` / `[USER_CONFIRM]` 標記，精確產生對應的按鈕 UI。

**工作量：** 30 分鐘（只改 prompt 文字）  
**效果：** 使用者一鍵操作取代手動打字，互動偵測準確率從 80% 提升到 99%+

---

### 總覽

| # | 改動 | 工作量 | 影響 | 風險 |
|---|------|--------|------|------|
| 1 | 移除 `pause` | 5 min | 消除子視窗卡住問題 | 極低 |
| 2 | Phase 1 拆步輸出 | 30 min | 使用者秒看摘要，不用乾等 | 低（只改 prompt） |
| 3 | 互動標記 | 30 min | 一鍵操作取代手動打字 | 低（LLM 偶爾漏標記） |

三個建議都只修改 prompt 或 Python script，不影響現有 tool YAML schema，**向下相容 CLI 使用**。

## English Version

### Suggestion 1: Remove `pause` from subprocess batch files

**Problem:**  
`sherlog_subprocess.py` and `displaydebugger_subprocess.py` generate batch files with `pause` at the end, causing child windows to wait for a keypress before closing. This is useful for manual CLI usage, but in automated / Extension scenarios, the child windows never close, blocking the entire analysis flow.

**Affected files:**
- `src/sherlog_subprocess.py` line 116
- `src/displaydebugger_subprocess.py` corresponding location

**Suggested fix:**  
Remove `pause`, or make it conditional via environment variable:
```batch
if "%GNAI_AUTOMATED%"=="" pause
```
This way CLI users can still inspect output manually, while the Extension sets `GNAI_AUTOMATED=1` to auto-close child windows.

**Effort:** 5 minutes  
**Impact:** Child windows close automatically; Bridge no longer needs taskkill workaround

---

### Suggestion 2: Split Phase 1 into two steps — output quick data first, run long tools after

**Problem:**  
The current `sighting_assistant.yaml` prompt causes the LLM to invoke all Phase 1 tools at once (including `sherlog_sync` and `displaydebugger`), then waits for **all of them to complete** before outputting the first line of response. Sherlog analysis can take 1–5 minutes, during which the user sees nothing but a loading spinner.

**Affected files:**
- `assistants/sighting_assistant.yaml` prompt section

**Suggested fix:**  
Explicitly split Phase 1 into two steps in the prompt:

- **Step A (quick, 10–20s):** Call `sighting_read_article`, `sighting_attachments`, `sighting_similarity_search`, `sighting_rag_search` → **immediately output a preliminary summary** (HSD title, category, component, attachment list, similar HSDs, DFD checklist)
- **Step B (long-running, 1–5min):** Call `sighting_sherlog_sync`, `sighting_gop_analyzer`, `sighting_displaydebugger` → append deep analysis results after completion

Add a strong directive in the prompt: `YOU MUST output the preliminary summary BEFORE calling Step B tools. DO NOT batch all tools together.`

**Effort:** 30 minutes (prompt text only, no tool code changes)  
**Impact:** Users see HSD basic info within 10–20 seconds instead of waiting for Sherlog to finish

---

### Suggestion 3: Add structured markers to interactive questions

**Problem:**  
When the LLM asks the user to make a selection (e.g., choose attachments to analyze, confirm DisplayDebugger execution), the response format is inconsistent. The Extension currently relies on regex heuristics to detect numbered lists or Yes/No questions, achieving ~80% accuracy. This occasionally generates wrong buttons or misses interactive prompts.

**Affected files:**
- `assistants/sighting_assistant.yaml` prompt section

**Suggested fix:**  
Require the LLM to wrap interactive questions with fixed markers in the prompt:

Multiple selection:
```
[USER_SELECT]
1) SystemScopeTool.zip
2) crash_dump.dmp
3) All
[/USER_SELECT]
```

Confirmation:
```
[USER_CONFIRM]
Would you like to run DisplayDebugger analysis on the display logs?
[/USER_CONFIRM]
```

The Extension detects `[USER_SELECT]` / `[USER_CONFIRM]` markers and precisely generates the corresponding button UI.

**Effort:** 30 minutes (prompt text only)  
**Impact:** One-click interaction replaces manual typing; detection accuracy improves from ~80% to 99%+

---

### Summary

| # | Change | Effort | Impact | Risk |
|---|--------|--------|--------|------|
| 1 | Remove `pause` | 5 min | Eliminates stuck child windows | Very low |
| 2 | Split Phase 1 output | 30 min | Users see summary instantly | Low (prompt-only) |
| 3 | Interaction markers | 30 min | One-click replaces typing | Low (LLM may occasionally miss markers) |

All three suggestions modify only prompt text or Python scripts. They do not affect existing tool YAML schemas and are **fully backward-compatible with CLI usage**.
